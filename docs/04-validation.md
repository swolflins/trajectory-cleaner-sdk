## 1. 验证概述

### 1.1 验证目的

本验证旨在评估轨迹清洗 SDK（设计见 [03-design.md](./03-design.md)）的实际效果，回答以下问题：

1. **去噪效果**：飞点检测的 Precision（精确率）/ Recall（召回率）/ F1 是否达标？
2. **伪静止识别**：静止抖动剔除是否准确，保留率是否符合预期？
3. **轨迹保真**：清洗后轨迹形态是否失真，里程误差是否可控？
4. **参数敏感性**：关键参数变化对效果的影响，最优参数组合是什么？
5. **性能**：在不同数据规模下的处理延迟与内存占用？
6. **双语言一致性**：Go 与 Java 实现结果是否对齐？

### 1.2 验证范围

| 维度 | 覆盖 |
|------|------|
| Stage | 精度过滤、伪静止状态机、异常检测（全部三阶段） |
| 数据集 | Seattle、GeoLife、合成数据集 |
| 指标 | Precision / Recall / F1 / RMSE / P95 / 保留率 |
| 参数 | 关键参数网格搜索 |
| 性能 | 1K / 10K / 100K / 1M 点规模 |
| 语言 | Go 与 Java 对拍 |

### 1.3 验证原则

1. **可复现**：固定随机种子、固定数据集版本、固定参数，结果可复现
2. **可对拍**：Go 与 Java 输入输出严格对比
3. **多维度**：不单一指标，综合评估去噪效果、保真度、性能
4. **诚实报告**：包括失败案例与边界情况

## 2. 验证方法论

### 2.1 验证流程总览

```
┌─────────────────────────────────────────────────────────────┐
│ 1. 准备数据集（真实 + 合成，含标注的异常点）                  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. 注入已知异常（飞点 / 静止抖动 / 漂移）                    │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 3. SDK 清洗处理（按参数组合执行）                            │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 4. 计算评估指标（Precision / Recall / F1 / RMSE / 保留率）   │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 5. 多维度分析（形态 / 分布 / 逐点 / 性能 / 敏感性）          │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 6. 参数调优（网格搜索 → 最优参数推荐）                       │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 异常注入方法

为获得 Ground Truth（真值），在干净轨迹上注入已知异常：

#### 2.2.1 飞点注入

- 在轨迹随机位置插入 N 个飞点
- 飞点经纬度偏离原点 D 米（D ∈ {100, 500, 1000, 5000}）
- 飞点时间戳保持原序列递增
- 记录飞点索引与偏离量作为真值

#### 2.2.2 静止抖动注入

- 选取轨迹中某段时间段，模拟设备静止
- 在真实位置周围半径 R 米（R ∈ {5, 10, 20, 30}）随机抖动
- 注入 M 个抖动点（M ∈ {10, 50, 100, 200}）
- 记录静止段起止索引与抖动半径作为真值

#### 2.2.3 漂移注入

- 选取连续 K 个点，整体偏移 Δ 米
- 偏移方向随机
- 模拟多径效应导致的持续漂移

#### 2.2.4 精度劣化注入

- 随机选取部分点，将 accuracy 设为高值（如 200 米）
- 用于验证 Stage 1 精度过滤

### 2.3 Ground Truth 标注

对于真实数据集（Seattle、GeoLife），采用以下方式获取标注：

1. **半自动标注**：先用 SDK 默认参数清洗，人工复核清洗结果，标注真实异常
2. **交叉验证**：多人独立标注，取一致结果
3. **可视化辅助**：轨迹可视化后人工识别明显飞点与静止段

### 2.4 对比基线

| 基线 | 说明 |
|------|------|
| 原始轨迹（不清洗） | 下界参考 |
| 仅速度约束 | 最简方法 |
| 仅 Z-score | 单方法 |
| Z-score + 伪静止 | 本项目完整管线 |
| IQR + 伪静止 | 备选方法 |
| 卡尔曼滤波（参考实现） | 平滑方法对比 |

## 3. 数据集说明

### 3.1 Seattle Dataset

**来源**：Microsoft Research

**内容**：西雅图地区 GPS 轨迹采集，含 75 条轨迹，约 10 万点

**特点**：
- 城市环境，含多径效应
- 采样频率 1 Hz
- 含已知飞点与静止段
- 提供参考绑路结果（用于里程误差评估）

**用途**：主要验证集，评估城市环境清洗效果

### 3.2 GeoLife Dataset

**来源**：Microsoft Research

**内容**：182 用户在北京的 GPS 轨迹，含 17000+ 条轨迹，总点数 2500 万+

**特点**：
- 多交通方式（步行 / 自行车 / 公交 / 驾车）
- 长时间跨度（数年）
- 多场景（城市 / 郊区 / 山区）
- 采样频率不统一（1-5 秒）

**用途**：大规模验证、多场景适应性测试

**子集**：抽取 1000 条轨迹（每交通方式 250 条）作为验证子集

### 3.3 合成数据集

**目的**：可控的 Ground Truth，用于精确指标计算

**生成方法**：

1. **基础轨迹**：按给定路径生成干净轨迹点
2. **注入异常**：按 2.2 节方法注入已知异常
3. **记录真值**：异常点索引、类型、参数

**合成数据集配置**：

| 数据集 | 轨迹数 | 点数 / 轨迹 | 飞点比例 | 静止段比例 | 用途 |
|--------|--------|------------|---------|-----------|------|
| synth-clean | 100 | 1000 | 0% | 0% | 基线（无异常） |
| synth-sparse | 100 | 1000 | 2% | 5% | 稀疏异常 |
| synth-dense | 100 | 1000 | 10% | 20% | 密集异常 |
| synth-long | 10 | 10000 | 5% | 10% | 长轨迹 |
| synth-static | 50 | 2000 | 0% | 50% | 静止为主 |
| synth-noise | 100 | 500 | 5% | 5% | 高噪声 |

### 3.4 数据集划分

| 用途 | 数据集 | 占比 |
|------|--------|------|
| 参数调优（训练） | synth-dense + Seattle 50% | 50% |
| 效果验证（测试） | synth-sparse + Seattle 50% + GeoLife 子集 | 50% |

## 4. 评估指标定义

### 4.1 分类指标（飞点检测）

| 指标 | 定义 | 公式 |
|------|------|------|
| TP（真阳性） | 正确识别的飞点数 | - |
| FP（假阳性） | 误杀的正常点数 | - |
| FN（假阴性） | 漏检的飞点数 | - |
| TN（真阴性） | 正确保留的正常点数 | - |
| Precision（精确率） | 检出的飞点中真实飞点占比 | TP / (TP + FP) |
| Recall（召回率） | 真实飞点中被检出的占比 | TP / (TP + FN) |
| F1 | 精确率与召回率的调和平均 | 2·P·R / (P + R) |

**目标**：Precision ≥ 0.95，Recall ≥ 0.90，F1 ≥ 0.92

### 4.2 回归指标（轨迹保真）

| 指标 | 定义 | 公式 |
|------|------|------|
| RMSE（均方根误差） | 清洗后点与真值点的位置误差 | sqrt(mean((x_pred - x_true)² + (y_pred - y_true)²)) |
| MAE（平均绝对误差） | 清洗后点与真值点的平均位置误差 | mean(|x_pred - x_true| + |y_pred - y_true|) |
| 里程误差 | 清洗后里程与真值里程的偏差 | |M_clean - M_true| / M_true |
| 保留率 | 输出点数 / 输入点数 | OutputCount / InputCount |

**目标**：RMSE < 10 米，里程误差 < 5%，保留率 60%-95%（场景相关）

### 4.3 性能指标

| 指标 | 定义 |
|------|------|
| 延迟 P50 | 50% 请求的处理延迟 |
| 延迟 P95 | 95% 请求的处理延迟 |
| 延迟 P99 | 99% 请求的处理延迟 |
| 吞吐 | 每秒处理点数 |
| 内存峰值 | 处理过程中的最大内存占用 |

### 4.4 静止识别指标

| 指标 | 定义 |
|------|------|
| 静止段识别率 | 正确识别的静止段 / 真实静止段 |
| 抖动点剔除率 | 剔除的抖动点 / 真实抖动点 |
| 误判静止率 | 误判为静止的运动段 / 总运动段 |

### 4.5 综合评分

为便于参数对比，定义综合评分（0-100）：

```
Score = 0.3 * F1 * 100
      + 0.2 * (1 - 里程误差) * 100
      + 0.2 * 保留率合理性评分
      + 0.15 * 静止识别率 * 100
      + 0.15 * 性能评分
```

其中保留率合理性评分：保留率在 [0.6, 0.95] 区间得满分，偏离扣分。

## 5. 参数调优实验设计

### 5.1 网格搜索参数

对以下关键参数进行网格搜索（其余参数保持默认）：

| 参数 | 搜索范围 | 步长 | 点数 |
|------|---------|------|------|
| `AccuracyThreshold` | [20, 30, 40, 50, 80, 100] | - | 6 |
| `MinStaticPoints` | [3, 5, 8, 10, 15] | - | 5 |
| `MinDisplacement` | [1, 2, 3, 5, 8] | - | 5 |
| `ResumeDisplacement` | [5, 10, 15, 20] | - | 4 |
| `ZThreshold` | [1.5, 2.0, 2.5, 3.0] | - | 4 |
| `IQRK` | [1.0, 1.5, 2.0, 2.5] | - | 4 |
| `CalibrationSize` | [30, 50, 100, 200] | - | 4 |
| `SlidingWindowSize` | [50, 100, 200, 500] | - | 4 |
| `GracePeriod` | [0, 2, 3, 5, 8] | - | 5 |
| `OutlierStrategy` | [replace, drop] | - | 2 |
| `DetectorMethod` | [zscore, iqr] | - | 2 |

**全网格规模**：6×5×5×4×4×4×4×4×5×2×2 = 1,228,800 组合

由于全网格过大，采用**分阶段搜索**策略：

### 5.2 分阶段搜索策略

#### 阶段 1：单参数敏感性分析

固定其余参数为默认值，逐个扫描单个参数，绘制参数-效果曲线，识别敏感参数。

```
对每个参数 p:
    for value in p.搜索范围:
        config = DefaultConfig()
        config.p = value
        result = evaluate(config, synth-dense)
        record(value, result.F1, result.里程误差, result.保留率)
```

#### 阶段 2：敏感参数联合搜索

根据阶段 1 结果，选取 Top-3 敏感参数做联合网格搜索（其余固定为阶段 1 最优值）。

候选敏感参数（预期）：
- `ZThreshold`（飞点检测灵敏度）
- `MinStaticPoints` + `MinDisplacement`（静止判定）
- `GracePeriod`（级联误报防护）

#### 阶段 3：场景化调优

针对不同场景（车载 / 步行 / 单车）分别调优，得到场景化推荐配置。

### 5.3 评估函数

每组参数在 synth-dense 数据集上评估，计算综合 Score（见 4.5 节），取 Score 最高的参数组合。

```
function evaluate(config, dataset):
    results = []
    for track in dataset:
        cleaned = pipeline.Process(track, config)
        metrics = computeMetrics(cleaned, track.groundTruth)
        results.append(metrics)
    aggregate = aggregateMetrics(results)
    score = computeScore(aggregate)
    return score
```

### 5.4 交叉验证

为防止过拟合 synth-dense，最优参数在 Seattle 与 GeoLife 子集上交叉验证：

```
best_config = gridSearch(synth-dense)
seattle_score = evaluate(best_config, Seattle_test)
geolife_score = evaluate(best_config, GeoLife_subset)
if seattle_score < threshold or geolife_score < threshold:
    # 过拟合，回退到次优参数
    ...
```

## 8. 双语言一致性验证

### 8.1 对拍测试设计

**目的**：验证 Go 与 Java 实现行为一致、数值结果对齐。

**方法**：
1. 准备固定输入数据集（合成 + 真实子集）
2. 分别用 Go 与 Java 处理，输出 JSON
3. 逐字段对比输出，允许浮点容差 1e-9

### 8.2 对拍测试用例

| 用例 | 描述 | 关注点 |
|------|------|--------|
| case-empty | 空轨迹 | 双方返回空 |
| case-single | 单点 | 透传一致 |

### 9.4 性能优化建议

基于基准测试的瓶颈定位，提出以下优化建议：

#### 9.4.1 已识别瓶颈

| 瓶颈 | 影响 | 根因 |
|------|------|------|
| `fmt.Sprintf` 格式化 | 每点 5/5 次分配，176 B/点 | 每次处理都格式化 reason 字符串 |
| `velocityWindow[1:]` 切片操作 | 隐含数组拷贝 | 滑动窗口用 slice 切片而非环形缓冲 |
| `percentile` 排序拷贝 | IQR 模式下每次校准都排序+拷贝 | `sortFloat64s` 每次创建新切片 |

#### 9.4.2 优化方向

| 优化项 | 预期收益 | 复杂度 | 优先级 |
|--------|---------|--------|--------|
| reason 字符串预分配/惰性格式化 | 分配 5→0，延迟 -15% | 低 | P0 |
| 环形缓冲替代 slice 切片 | 减少 GC 压力 | 中 | P1 |
| 有序结构维护（IQR 模式） | 校准耗时 -50% | 中 | P2 |
| 浮点运算批处理向量化 | 延迟 -10% | 高 | P3 |

#### 9.4.3 容量规划参考

### C.4 ClickHouse 存储估算

#### 表结构

```sql
CREATE TABLE trajectory_points (
    device_id   UInt64,
    ts          DateTime64(3),
    lat         Float64 CODEC(Gorilla),
    lng         Float64 CODEC(Gorilla),
    accuracy    Float32,
    speed       Float32,
    heading     Float32
) ENGINE = MergeTree
PARTITION BY toYYYYMM(ts)
ORDER BY (device_id, ts)
SETTINGS index_granularity = 8192;
```

#### 架构优势

| 特性 | PostgreSQL | ClickHouse |
|------|-----------|------------|
| 存储格式 | 行存 (Heap) | 列存 (MergeTree) |
| 索引类型 | B-tree（每行一个条目） | 稀疏索引（每 8192 行一个条目） |
| 压缩 | 无（TOAST 仅超大字段） | LZ4 默认 / ZSTD 可选 |
| 列编码 | 无 | DoubleDelta（时间戳）、Gorilla（浮点） |
| 压缩比 | 1:1 | 5-30x（取决于列类型和 codec） |

> ClickHouse 稀疏索引每 8192 行仅存一个条目，索引体积远小于 B-tree；列存格式使相似值连续存储，LZ4 可达 5x 压缩，ZSTD 可达 10x 以上。ZSTD 压缩比 LZ4 小约 30%，但解压慢 2-3 倍，适合冷数据列。

#### 单行存储拆解

| 组成部分 | LZ4 模式 | ZSTD 模式 | 说明 |
|----------|----------|-----------|------|
| 原始列数据 | 52 B | 52 B | 7 列原始值 |
| 列存压缩后 | ~10 B | ~4 B | LZ4 ~5x，ZSTD ~10x |
| 稀疏索引均摊 | ~0.01 B | ~0.01 B | 1/8192 |
| marks + 元数据 | ~0.5 B | ~0.5 B | 列元数据 + granule 标记 |
| **总计** | **~11 B/行** | **~5 B/行** | |

#### 累计存储

| 时间 | 记录数 | ClickHouse LZ4 | ClickHouse ZSTD | vs PostgreSQL |
|------|--------|---------------|----------------|--------------|
| 1 天 | 3,456 万 | 0.37 GB | 0.17 GB | -97% / -99% |
| 1 个月 | 10.37 亿 | 11 GB | 5 GB | -94% / -97% |
| 6 个月 | 62.2 亿 | 55 GB | 26 GB | -95% / -98% |
| 12 个月 | 126.1 亿 | **110 GB** | **53 GB** | -95% / -97% |

|------|---------|------------|---------|----------|

### C.6 存储方案选型建议

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 通用 OLAP 分析 | ClickHouse (LZ4) | 列存查询快，压缩比高，ZSTD 用于冷数据 |
| 已有 PostgreSQL 基础设施 | TimescaleDB | 无需引入新组件，兼容 SQL 生态 |
| 需要事务/点更新 | PostgreSQL | 行存支持 UPDATE/DELETE，ClickHouse 不擅长 |
| 成本敏感 + 大规模 | ClickHouse (ZSTD) | 53 GB/年，云费用仅 $28/年 |
| 混合场景 | PG (热) + ClickHouse (冷) | 热数据 PG 供实时查询，冷数据 ClickHouse 供分析 |

### C.7 推荐架构

结合清洗程序性能与 ClickHouse 存储优势，推荐以下部署架构：

```
设备上报 (400台 × 1Hz)
    │
    ▼
Kafka / 消息队列 (缓冲 + 削峰)
    │
    ▼
清洗程序 (Go SDK)          ← CPU 0.03%, 内存 192KB, 单核可处理
    │
    ▼
ClickHouse (MergeTree)      ← 53 GB/年 (ZSTD), $28/年存储成本
    │
    ▼
BI / 查询 / 分析
```

- 清洗程序作为 Kafka 消费者部署，单实例即可处理 400 台设备的全部流量
- ClickHouse 按 `toYYYYMM(ts)` 分区，历史数据自动压缩
- 热数据（近 7 天）不压缩保查询性能，冷数据自动 TTL 压缩
- 设备维度查询用 B-tree 排序键（ORDER BY device_id, ts），纯时间范围查询用 BRIN 索引
